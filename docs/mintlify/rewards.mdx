# Rewards

> **Status: disabled in production.** No `RewardVault` is deployed (`rewardVault == 0`), so **no reward leg
> is carved on any trade** — trades behave exactly as if this page didn't exist. The mechanism below is
> documented for completeness and can be enabled later by deploying a vault and pointing the router at it.

On top of the existing fee model, **every trade pays two additive 0.25% reward legs**. A **buy** routes 0.25% of its ETH to that coin's **Traders** pot; a **sell** routes 0.25% of its ETH-out to that coin's **Holders** pot. The legs are carved *on top* of the project's swap fee and forwarded to the `RewardVault` as raw ETH — they never touch the platform or creator escrows.

> Two 0.25% legs, keyed by `(coin, epoch, side)`. Buy → **Traders** (rewards net accumulation). Sell → **Holders** (rewards size × time). Rewards are per-coin: a coin's legs only ever pay that coin's own traders and holders.

On every trade `PadRouter` calls `RewardVault.accrue{value}(coin, side)`. The calls are wrapped in `try/catch`, so a paused or misconfigured vault can **never revert a trade** — on failure the leg becomes that coin's floor instead. The legs are **off** whenever no vault is set (`rewardVault == 0`), and trades behave exactly as before.

## How it's scored

The chain custodies the ETH; an off-chain indexer (the **poster**) computes each wallet's share per epoch. Two sides, two rules:

| Side | Funded by | Weight |
|------|-----------|--------|
| **Traders** (`0`) | the 0.25% buy leg | `max(0, net token accumulation in the epoch)` — buys − sells. A round-tripper nets ~0. |
| **Holders** (`1`) | the 0.25% sell leg | **balance-seconds** — `∫ balance dt` over the epoch, reconstructed from trades and clamped ≥ 0. |

Each wallet's payout is `floor(pot · weight / Σ weight)`, so the sum of allocations can never exceed the pot. The floored wei remainder is left unclaimed and later swept to the coin's floor.

The exact scoring rules are frozen in a spec string (`ALGO_SPEC`); its keccak is posted on-chain as `algoHash` with every root, so anyone can recompute the allocation and prove a mismatch.

## Epochs & roots

Rewards settle in fixed epochs. After an epoch ends and a finality delay passes, the poster posts **one Merkle root** covering every coin's leaves for that epoch:

```solidity
function postRoot(uint256 epoch, bytes32 root, bytes32 algoHash, string uri) // poster only
```

- Allowed only once the epoch has fully ended **and** `finalityDelay` has elapsed past its end (reorg safety).
- Write-once per epoch — unless the epoch was vetoed, in which case a corrected root may replace it.
- `uri` pins the full leaf set (IPFS/Arweave) so the root is independently verifiable.

Windows are **snapshotted into the epoch at `postRoot`**, so a later admin change can never retroactively move an in-flight epoch's veto or sweep boundary.

| Window | Production default | Meaning |
|--------|--------------------|---------|
| `EPOCH` | **7 days** | Reward epoch length (`block.timestamp / EPOCH`). |
| `finalityDelay` | **1 day** | A root may only be posted this long after the epoch ends. |
| `challengeWindow` | **2 days** | After `postRoot`: the guardian-veto window; **claims open when it ends**. |
| `claimWindow` | **30 days** | Grace period after claims open before an unclaimed remainder can be swept to the floor. |

## Claiming

Users claim their own leaf and pay their own gas:

```solidity
// side: 0 = Traders, 1 = Holders
function claim(uint256 epoch, address coin, uint8 side, uint256 amount, bytes32[] proof)
```

The claim is a pure Merkle verify + a **per-side conservation cap** + an ETH transfer. It's allowed once the challenge window has passed and the epoch isn't vetoed. The cap (`Σ claims ≤ pot` for each `(coin, epoch, side)`) guarantees total payouts can never exceed what actually accrued.

The leaf binds `(epoch, coin, side, user, amount)` and is double-hashed to match OpenZeppelin's `StandardMerkleTree` — the off-chain `leafHash` produces the identical value:

```solidity
keccak256(bytes.concat(keccak256(abi.encode(
  uint256 epoch, address coin, uint8 side, address user, uint256 amount))))
```

Because `amount` is baked into the leaf, there's nothing for the contract to compute — it just verifies the proof, checks the cap, and pays.

## Guardian veto & sweep

- **Veto.** A `guardian` (a multisig) can `veto(epoch)` a bad root — but **only inside the challenge window, before claims open**. A veto can only *block* finalization; it never moves funds. The poster then re-posts a corrected root, restarting the window.
- **Sweep.** After the claim window closes, anyone can `sweep(epoch, coin)`: the unclaimed remainder of both sides is donated into that coin's Bond Bounty buy-wall via the router's audited floor plumbing. Dead rewards become permanent price floor for the same community; funds can only ever go to the coin's *own* floor.

## Trust model

The poster is an off-chain operator you trust to score correctly. What bounds it:

- **Per-side conservation cap (the fuse).** Even a fully malicious or buggy root can, worst case, misallocate **one coin's own accrued fees among that coin's own addresses for one side** — never cross-coin, never cross-side, never protocol theft.
- **Frozen `algoHash` + pinned leaf set.** Anyone can recompute the allocation and prove a mismatch.
- **Challenge window + guardian veto.** A provably-wrong root can be blocked before any claim is paid.

So the on-chain surface is minimal by design: the chain does the one thing it must do trustlessly — custody the ETH and cap what can ever leave.

## From the pad SDK (`assets/wallet.js`)

The browser reads claimable rewards + proofs from the [indexer API](api.md) and spends a plain `claim()` per leaf.

```js
import * as Pad from "./assets/wallet.js";

// what this wallet can claim now, plus what's still accruing this epoch
const { claimable, pending, totals } = await Pad.rewards(myAddress);
// claimable[i] = { coin, side, epoch, amount, proof, eth, name, sym, ... }

// claim one leaf (a row from claimable)
await (await Pad.claimReward(claimable[0])).wait();

// or sweep every available leaf (one signature each)
const n = await Pad.claimAllRewards();
```

`pending` is a live provisional estimate against the current, not-yet-finalized epoch (no proof yet); `claimable` only lists epochs whose root is already on-chain.
